# Распределённый фаззинг

[Общий план](FUZZING_PLAN.md) | [Phase 2](FUZZING_PHASE2.md)

## Зачем

Одна машина с 8 ядрами исчерпывает coverage TL-парсера за минуты.
Для `pool.cpp` (Phase 2) пространство состояний ~2^64 — горизонтальное масштабирование
даёт линейный рост итераций, но без координации машины дублируют работу.

Цель: **не просто в ширину, а умнее** — каждая машина исследует свою часть пространства.

---

## Уровень 1 — corpus sync (просто, без координации)

Каждая машина фаззит независимо, периодически обменивается корпусом:

```bash
# на каждой машине, каждый час:
rsync -a machine2:~/tonGraph/simulation/corpus_fuzz_tl/ simulation/corpus_fuzz_tl/
rsync -a machine3:~/tonGraph/simulation/corpus_fuzz_tl/ simulation/corpus_fuzz_tl/
./build/test/consensus/fuzz_tl -merge=1 corpus_merged/ simulation/corpus_fuzz_tl/
mv corpus_merged/* simulation/corpus_fuzz_tl/
```

**Плюсы:** нет инфраструктуры, работает сразу.
**Минусы:** дублирование — машины независимо находят одни и те же пути.

---

## Уровень 2 — разделение по стратегиям (рекомендуется первым)

Каждая машина специализируется на своей стратегии мутации. Корпус мержится раз в час.

| Машина | Стратегия | Флаги |
|---|---|---|
| A | Короткие мутации, широкое покрытие | `-mutate_depth=1 -max_len=64` |
| B | Глубокие цепочки мутаций | `-mutate_depth=8` |
| C | Только targeted seeds (gen_targeted_corpus) | стартовый корпус из `corpus_fuzz/` |
| D | WAL crash injection (Phase 2, Шаг 3) | отдельный harness с MockDb::crash |

Машины A–C шарят `corpus_fuzz_tl/`, машина D — `corpus_fuzz_pool/`.

## Уровень 3 — distributed directed fuzzing (координатор)

Идея из CollabFuzz/Cupid: один координатор назначает каждому воркеру свою цель —
конкретную ветку кода которую он должен покрыть. Воркеры не дублируют уже покрытое.

```
Координатор (Redis)
    ├── хранит граф покрытия (какие базовые блоки уже покрыты)
    ├── назначает воркеру: "твоя цель — базовый блок 0x1234"
    └── воркер присылает новые пути → координатор обновляет граф

Воркеры
    ├── LLVMFuzzerCustomMutator: мутирует в сторону назначенной цели
    └── по достижении цели — запрашивают следующую у координатора
```

**Реализация:** libFuzzer + `LLVMFuzzerCustomMutator` + Redis как очередь целей.
**Когда применять:** если Уровень 2 за 72 часа не даёт новых крашей.

## Уровень 4 — distance-guided с общей vector DB

Из плана Phase 2: Faiss/hnswlib на отдельной машине хранит "эталонные опасные состояния"
из известных PoC. Воркеры присылают snapshot состояния pool.cpp, получают сигнал
"насколько близко к нарушению".

```
Vector DB (Faiss/hnswlib) — отдельная машина
    ├── хранит эталонные опасные состояния (notarize_weight, voted_notar, ...)
    └── API: cosine_similarity(snapshot) → float [0, 1]

Воркеры
    ├── после каждого LLVMFuzzerTestOneInput: снимают snapshot состояния
    ├── запрашивают similarity у vector DB
    └── LLVMFuzzerCustomMutator: мутирует в сторону уменьшения расстояния
```

**Это градиентный спуск в пространстве протокольных решений**, а не случайный поиск.
**Когда применять:** если Уровни 1–3 не дали SAFETY VIOLATION.

## Практический порядок для нашего проекта

```
Сейчас:     1 машина, fuzz_tl (Шаг 1) — плато за минуты, достаточно
            ↓
После Шага 2 (fuzz_pool):
            Уровень 1 — corpus sync между 2–3 машинами
            + Уровень 2 — разные стратегии мутации
            ↓ (если 72ч без крашей)
            Уровень 3 — координатор на Redis
            ↓ (если исчерпано)
            Уровень 4 — vector DB
```
